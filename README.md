# 端口健康监测 · Port Health Monitor

一个轻量的本地端口/服务健康监测小服务。定时探测你注册的本地端口是否可连通，按 **10 秒** 粒度记录健康历史，并提供一个 beszel 风格的浅色现代化 Web 界面。

- 🟢 **健康探测**：每 10 秒对 `127.0.0.1:<端口>` 发起一次 TCP 连接探测，记录在线/离线。
- 🗂 **历史记录**：紧凑的追加式二进制时序存储，1 个月的数据每个服务约 1.3 MB。
- 🖥 **Web 前端**：挂在 `3333` 端口，浅色简约风格，右上角切换时间范围（近 10 分钟 / 1 小时 / 1 天 / 7 天 / 1 月，默认 1 天），并自动调整展示粒度。
- ⏱ **运行时长**：实时显示服务自「上次启动」或「上次终止」以来的持续时长，精确到秒；注册前无数据则置灰为「数据不可用」。
- 🖱 **卡片交互**：点击服务名称即可跳转其公网网址（名称 + ↗ 一体高亮）；点击服务根目录一键复制路径到剪贴板；卡片可编辑、删除。
- 🔎 **端口速查**：一键弹出当前本机**正在监听的 TCP 端口**表格，直接从进程命令行读出各端口在跑什么服务；任意列可升/降序排序；已注册的端口显示对应服务名，未注册的可一键注册（自动带入端口号，并把读取到的名称作为灰色默认值，可覆盖），也可**忽略**——把系统进程、临时端口等从「未注册」提醒中隐藏，并可附上一条不强制的备注，随时取消。
- 🔔 **通知与告警**：右上角**铃铛/齿轮**两个图标各弹出一个右侧抽屉。**设置**里关联 Telegram Bot（Token / Chat ID，可开关、可发测试消息；SMTP 邮件 UI 已预留）；**通知**里添加告警规则——① 端口状态实时变更、② 某状态持续超过 N 秒、③ 发现新占用端口（既未注册也未忽略）。前两类可关联指定端口或全部端口，三类均可多选推送通道（现为 Telegram，邮件预留）。
- 🤖 **Telegram 交互**：机器人菜单可**概览所有端口**（名称 + 状态 + 外网链接），或选定某端口查看 **1 分钟 / 10 分钟 / 1 小时 / 12 小时 / 1 天 / 7 天**的在线/离线时长与在线率。
- 🚫 **无需性能分析**，无需自动发现端口，一切服务通过网页端手动注册。
- 📦 **零第三方依赖**，仅使用 Node.js 内置模块（Telegram 走内置 `https`）。

## 运行

需要 Node.js ≥ 18。

```bash
npm start
# 或
node src/server.js
```

然后打开 <http://localhost:3333>。

## 开机自启（systemd）

仓库自带 `deploy/port-health-monitor.service`。安装后随开机自动启动，崩溃自动重启：

```bash
sudo cp deploy/port-health-monitor.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now port-health-monitor.service
```

常用命令：

```bash
systemctl status port-health-monitor     # 查看状态
systemctl is-enabled port-health-monitor # 是否开机自启（enabled）
sudo systemctl restart port-health-monitor  # 改动代码后重启以生效
journalctl -u port-health-monitor -f      # 跟踪日志
```

> 本机已安装并启用该服务（`enabled` + `running`），无需重复操作。

## 通过反向代理 / Cloudflare Tunnel 访问

前端资源（JS/CSS/HTML）一律以 `Cache-Control: no-store` 返回，且 `index.html` 会
按文件修改时间自动为静态资源追加版本号（`/app.js?v=<mtime>`），避免 Cloudflare 等
代理按扩展名缓存旧版本。改动前端后普通刷新即可生效，无需清缓存。
（极少数情况下若在 Cloudflare 设了 “Cache Everything” 页面规则，需手动 Purge 一次。）

## 使用

1. 点击左上角标题旁的「**+ 注册服务**」按钮，在弹窗中填写：
   - **服务名称**（必填）
   - **端口号**（必填，1–65535）
   - **服务根目录**（可选，用于展示与一键复制）
   - **公网网址**（可选，仅用于页面跳转，**不会被监测**）
2. 注册后即开始每 10 秒探测一次，历史条会逐步填充。
3. 每张服务卡片：
   - 状态圆点（绿=在线 / 红=离线 / 灰=未知）+ 服务名 + 端口徽章；
   - 点击**服务名**跳转公网网址（若已填）；右上角显示所选范围内的**在线率**，以及**编辑 ✎ / 删除 ✕**；
   - 中部为全宽无缝**状态历史条**，鼠标悬停某一格可查看该时段的时间区间、在线率与探测次数；
   - 底部一行：**📁 服务根目录**（点击复制路径，页面底部弹出「已复制」提示）、**🔗 公网网址**，以及右对齐的**运行/中断时长**（每秒实时跳动，无数据则灰显「数据不可用」）。
4. 点击标题旁的「**🔎 端口速查**」可查看本机当前正在监听的所有 TCP 端口：
   - 每行显示**端口**、**监听地址**、**服务 / 命令**（直接读自进程命令行）以及**注册情况**；
   - 点击任意**表头**即可按该列升序/降序排序；
   - 已在本工具注册的端口显示绿色的服务名徽章；未注册的端口右侧提供「**注册**」按钮，点击后叠加打开注册弹窗，自动带入端口号，并把读取到的服务名作为**灰色默认值**填入（留空即用它，填写则覆盖）；
   - 「注册」旁另有「**忽略**」按钮，用于把系统进程、临时/调试端口等从未注册提醒中隐藏——弹窗里可填一条**可选**备注（不强制），忽略后该行显示灰色「已忽略」徽章及备注，点「**取消忽略**」即可恢复。
   - 读取依赖系统的 `ss` 命令；为能显示其它用户（如 root）进程的名称，会优先尝试 `sudo -n ss`，无免密 sudo 时自动回退到普通 `ss`（仍列出全部端口，仅本用户进程可见名称）。
   - 命令行中的敏感参数（`--token` / `--password` / `--api-key` / `SECRET=…` / JWT 等）在返回前会被自动遮蔽为 `***`，避免在页面上泄露密钥。
5. 右上角切换时间范围，历史条的聚合粒度会自动变化：

   | 范围 | 时间窗口 | 每格粒度 |
   |------|---------|---------|
   | 近 10 分钟 | 10 min | 10 秒 |
   | 近 1 小时 | 1 h | 1 分钟 |
   | 近 1 天（默认）| 24 h | 10 分钟 |
   | 近 7 天 | 7 d | 1 小时 |
   | 近 1 月 | 30 d | 6 小时 |

## 通知与告警

右上角时间滑块左侧有两个图标按钮，点击各自从右侧滑出一个抽屉。

### 🔔 通知抽屉——告警规则

点击「**添加告警**」，选择类型：

| 类型 | 触发条件 | 可配置项 |
|------|---------|---------|
| **端口状态变更** | 10 秒探测发现端口由可用↔不可用翻转时立即推送 | 关联端口（全部/指定）、方向（变为可用/不可用/两者） |
| **状态持续时长** | 端口在某状态**持续超过**设定时长后推送一次（同一次持续只推一次） | 关联端口（全部/指定）、状态（可用/不可用）、时长（秒，含快捷预设） |
| **新占用端口** | 端口速查每 20 秒扫描，发现**既未注册也未忽略**的新监听端口时推送 | —（启动时的既有端口作为基线，不会补推） |

每条规则可**多选推送通道**（当前为 Telegram，SMTP 邮件已预留多选/多收件人位）；规则卡片可随时开关、编辑、删除。

### ⚙️ 设置抽屉——通知通道

1. 用 Telegram 找 **@BotFather** 创建一个 Bot，拿到 **Bot Token**。
2. 给你的 Bot 发送任意一句话，再发送 `/id`，它会回复本对话的 **Chat ID**。
3. 在设置抽屉里填入 Token 与 Chat ID，打开开关并「**保存**」；可先点「**发送测试消息**」验证连通。
   - Token 只存于服务端，页面只回显「是否已设置」，不会明文返回；修改时留空即保留原值。

### 🤖 Telegram Bot 交互

开启后，机器人会注册命令菜单（对话框旁的 **Menu** 按钮展开），并支持内联按钮导航：

- `/menu`——打开菜单：**📊 概览所有端口** / **🔍 查看指定端口**；
- **概览**：逐个列出服务的状态圆点、名称（附外网链接）与端口，并汇总在线/离线数量；
- **查看指定端口**：先选端口，再选 **1 分钟 / 10 分钟 / 1 小时 / 12 小时 / 1 天 / 7 天**，返回该时段的在线时长、离线时长与在线率；
- `/overview` 直接概览、`/id` 查询 Chat ID。
- 出于隐私，数据类命令仅响应已绑定的 Chat ID（`/id` 除外，便于获取自己的 ID）。

## 时间范围与粒度

历史条的每一格聚合了该时间段内的多次探测：

- **绿色**：该时段全部在线
- **红色**：该时段全部离线
- **橙色**：部分在线（有过波动）
- **灰色**：该时段无探测数据

## 数据存储

- `data/services.json` — 服务元数据与当前状态（当前状态、状态起始时间等）。
- `data/checks/<id>.bin` — 每个服务的健康历史，追加式二进制日志，每条记录 5 字节：
  - 4 字节：Unix 时间戳（秒，小端 uint32）
  - 1 字节：状态（1 = 在线，0 = 离线）
- 超过最长时间窗口（1 月 + 缓冲）的旧记录会被定期清理。

删除服务会同时删除其历史文件。

## HTTP API

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/data?range=1d` | 仪表盘一次性获取的全部数据（服务列表 + 聚合历史条）。`range` ∈ `10min,1h,1d,7d,1m` |
| `GET` | `/api/ports` | 当前正在监听的 TCP 端口列表（端口、监听地址、进程/命令、是否已注册及注册名、是否已忽略及备注） |
| `GET` | `/api/services` | 仅服务元数据列表 |
| `POST` | `/api/services` | 注册服务，body：`{name, port, rootDir?, publicUrl?}` |
| `PUT` | `/api/services/:id` | 修改服务元数据 |
| `DELETE` | `/api/services/:id` | 删除服务及其历史 |
| `POST` | `/api/ports/ignore` | 忽略某端口，body：`{port, note?}`（备注可选；重复提交更新备注） |
| `DELETE` | `/api/ports/ignore/:port` | 取消忽略某端口 |
| `GET` | `/api/settings` | 通知通道配置（Token/密码等密钥以布尔位回显，不明文返回） |
| `PUT` | `/api/settings` | 更新通道配置，body：`{telegram:{enabled,token?,chatId}, smtp:{…}}`（密钥留空则保留原值） |
| `POST` | `/api/settings/telegram/test` | 发送一条测试消息，可带 `{token?, chatId?}` 在保存前测试 |
| `GET` | `/api/rules` | 全部告警规则 |
| `POST` | `/api/rules` | 新增告警规则 |
| `PUT` | `/api/rules/:id` | 修改告警规则 |
| `DELETE` | `/api/rules/:id` | 删除告警规则 |

## 项目结构

```
src/
  server.js    HTTP 服务、路由、聚合、静态资源、通知接线
  monitor.js   每 10 秒的探测循环与 TCP 端口检测（探测后回调告警引擎）
  ports.js     端口速查：读取正在监听的 TCP 端口及其进程/命令（ss + /proc）
  notify.js    告警引擎：状态变更 / 持续时长 / 新占用端口（20s 扫描）三类规则求值与分发
  telegram.js  Telegram Bot：内置 https 发消息 + 长轮询交互（概览、指定端口时段统计）
  store.js     无依赖的持久化层（元数据 + 二进制时序 + 设置/规则）
  ranges.js    时间范围与粒度定义
public/
  index.html   页面结构（含通知/设置抽屉与规则编辑器）
  styles.css   beszel 风格浅色主题
  app.js        前端逻辑（渲染、计数器、tooltip、toast、注册/编辑/删除、复制、端口速查、忽略端口、通知/设置抽屉）
deploy/
  port-health-monitor.service   systemd 开机自启单元
data/          运行时数据（不纳入版本控制）
  services.json  已注册服务元数据 + 当前状态
  ignores.json   端口速查中被忽略的端口及备注
  settings.json  通知通道配置（Telegram / SMTP；含密钥，勿外泄）
  rules.json     告警规则
  checks/        每个服务的二进制健康历史日志
```

## 设计取舍

- **为何不用 SQLite**：目标环境缺少原生编译工具链，且固定 10 秒粒度的时序数据用紧凑二进制日志已足够高效，因此选择零依赖方案，避免安装/编译问题。
- **时钟对齐**：接口返回服务端时间，前端据此校正本地时钟偏移，保证「运行时长」计数准确。
