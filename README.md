# 端口健康监测 · Port Health Monitor

一个轻量的本地端口/服务健康监测小服务。定时探测你注册的本地端口是否可连通，按 **10 秒** 粒度记录健康历史，并提供一个 beszel 风格的浅色现代化 Web 界面。

- 🟢 **健康探测**：每 10 秒对 `127.0.0.1:<端口>` 发起一次 TCP 连接探测，记录在线/离线。
- 🗂 **历史记录**：紧凑的追加式二进制时序存储，1 个月的数据每个服务约 1.3 MB。
- 🖥 **Web 前端**：挂在 `3333` 端口，浅色简约风格，右上角切换时间范围（近 10 分钟 / 1 小时 / 1 天 / 7 天 / 1 月，默认 1 天），并自动调整展示粒度。
- ⏱ **运行时长**：实时显示服务自「上次启动」或「上次终止」以来的持续时长，精确到秒；注册前无数据则置灰为「数据不可用」。
- 🖱 **卡片交互**：点击服务名称即可跳转其公网网址（名称 + ↗ 一体高亮）；点击服务根目录一键复制路径到剪贴板；卡片可编辑、删除。
- 🔎 **端口速查**：一键弹出当前本机**正在监听的 TCP 端口**表格，直接从进程命令行读出各端口在跑什么服务；任意列可升/降序排序；已注册的端口显示对应服务名，未注册的可一键注册（自动带入端口号，并把读取到的名称作为灰色默认值，可覆盖）。
- 🚫 **无需性能分析**，无需自动发现端口，一切服务通过网页端手动注册。
- 📦 **零第三方依赖**，仅使用 Node.js 内置模块。

## 运行

需要 Node.js ≥ 18。

```bash
npm start
# 或
node src/server.js
```

然后打开 <http://localhost:3333>。

## 开机自启（systemd）

仓库自带 `deploy/port-health-monitor.service`。安装后随开机自动启动，崩溃自动重启：

```bash
sudo cp deploy/port-health-monitor.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now port-health-monitor.service
```

常用命令：

```bash
systemctl status port-health-monitor     # 查看状态
systemctl is-enabled port-health-monitor # 是否开机自启（enabled）
sudo systemctl restart port-health-monitor  # 改动代码后重启以生效
journalctl -u port-health-monitor -f      # 跟踪日志
```

> 本机已安装并启用该服务（`enabled` + `running`），无需重复操作。

## 通过反向代理 / Cloudflare Tunnel 访问

前端资源（JS/CSS/HTML）一律以 `Cache-Control: no-store` 返回，且 `index.html` 会
按文件修改时间自动为静态资源追加版本号（`/app.js?v=<mtime>`），避免 Cloudflare 等
代理按扩展名缓存旧版本。改动前端后普通刷新即可生效，无需清缓存。
（极少数情况下若在 Cloudflare 设了 “Cache Everything” 页面规则，需手动 Purge 一次。）

## 使用

1. 点击左上角标题旁的「**+ 注册服务**」按钮，在弹窗中填写：
   - **服务名称**（必填）
   - **端口号**（必填，1–65535）
   - **服务根目录**（可选，用于展示与一键复制）
   - **公网网址**（可选，仅用于页面跳转，**不会被监测**）
2. 注册后即开始每 10 秒探测一次，历史条会逐步填充。
3. 每张服务卡片：
   - 状态圆点（绿=在线 / 红=离线 / 灰=未知）+ 服务名 + 端口徽章；
   - 点击**服务名**跳转公网网址（若已填）；右上角显示所选范围内的**在线率**，以及**编辑 ✎ / 删除 ✕**；
   - 中部为全宽无缝**状态历史条**，鼠标悬停某一格可查看该时段的时间区间、在线率与探测次数；
   - 底部一行：**📁 服务根目录**（点击复制路径，页面底部弹出「已复制」提示）、**🔗 公网网址**，以及右对齐的**运行/中断时长**（每秒实时跳动，无数据则灰显「数据不可用」）。
4. 点击标题旁的「**🔎 端口速查**」可查看本机当前正在监听的所有 TCP 端口：
   - 每行显示**端口**、**监听地址**、**服务 / 命令**（直接读自进程命令行）以及**注册情况**；
   - 点击任意**表头**即可按该列升序/降序排序；
   - 已在本工具注册的端口显示绿色的服务名徽章；未注册的端口右侧提供「**注册**」按钮，点击后叠加打开注册弹窗，自动带入端口号，并把读取到的服务名作为**灰色默认值**填入（留空即用它，填写则覆盖）。
   - 读取依赖系统的 `ss` 命令；为能显示其它用户（如 root）进程的名称，会优先尝试 `sudo -n ss`，无免密 sudo 时自动回退到普通 `ss`（仍列出全部端口，仅本用户进程可见名称）。
   - 命令行中的敏感参数（`--token` / `--password` / `--api-key` / `SECRET=…` / JWT 等）在返回前会被自动遮蔽为 `***`，避免在页面上泄露密钥。
5. 右上角切换时间范围，历史条的聚合粒度会自动变化：

   | 范围 | 时间窗口 | 每格粒度 |
   |------|---------|---------|
   | 近 10 分钟 | 10 min | 10 秒 |
   | 近 1 小时 | 1 h | 1 分钟 |
   | 近 1 天（默认）| 24 h | 10 分钟 |
   | 近 7 天 | 7 d | 1 小时 |
   | 近 1 月 | 30 d | 6 小时 |

## 时间范围与粒度

历史条的每一格聚合了该时间段内的多次探测：

- **绿色**：该时段全部在线
- **红色**：该时段全部离线
- **橙色**：部分在线（有过波动）
- **灰色**：该时段无探测数据

## 数据存储

- `data/services.json` — 服务元数据与当前状态（当前状态、状态起始时间等）。
- `data/checks/<id>.bin` — 每个服务的健康历史，追加式二进制日志，每条记录 5 字节：
  - 4 字节：Unix 时间戳（秒，小端 uint32）
  - 1 字节：状态（1 = 在线，0 = 离线）
- 超过最长时间窗口（1 月 + 缓冲）的旧记录会被定期清理。

删除服务会同时删除其历史文件。

## HTTP API

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/data?range=1d` | 仪表盘一次性获取的全部数据（服务列表 + 聚合历史条）。`range` ∈ `10min,1h,1d,7d,1m` |
| `GET` | `/api/ports` | 当前正在监听的 TCP 端口列表（端口、监听地址、进程/命令、是否已注册及注册名） |
| `GET` | `/api/services` | 仅服务元数据列表 |
| `POST` | `/api/services` | 注册服务，body：`{name, port, rootDir?, publicUrl?}` |
| `PUT` | `/api/services/:id` | 修改服务元数据 |
| `DELETE` | `/api/services/:id` | 删除服务及其历史 |

## 项目结构

```
src/
  server.js    HTTP 服务、路由、聚合、静态资源
  monitor.js   每 10 秒的探测循环与 TCP 端口检测
  ports.js     端口速查：读取正在监听的 TCP 端口及其进程/命令（ss + /proc）
  store.js     无依赖的持久化层（元数据 + 二进制时序）
  ranges.js    时间范围与粒度定义
public/
  index.html   页面结构
  styles.css   beszel 风格浅色主题
  app.js        前端逻辑（渲染、计数器、tooltip、toast、注册/编辑/删除、复制、端口速查）
deploy/
  port-health-monitor.service   systemd 开机自启单元
data/          运行时数据（不纳入版本控制）
```

## 设计取舍

- **为何不用 SQLite**：目标环境缺少原生编译工具链，且固定 10 秒粒度的时序数据用紧凑二进制日志已足够高效，因此选择零依赖方案，避免安装/编译问题。
- **时钟对齐**：接口返回服务端时间，前端据此校正本地时钟偏移，保证「运行时长」计数准确。
